Why this, and the hard requirements

Why. The findings registry that drives delta synthesis is only ~12 days deep because the cold-start seed ran over a thin, flawed input set. We rebuild from a real historical foundation by reprocessing raw article history through today's pipeline (not importing old findings). That exposed the gap: ingest is still file-based (fetch → substack_rss_raw.json → bundles → extraction) while extraction already writes the DB. The front half must become DB-native.

  • Requirement 1 — claim-level traceability, in the DB. Every finding traces, via the DB, to the synthesis/adjudication run that produced it → to the specific claims it rests on (with the "why") → to the article extraction → to the verbatim fetched feed. End state is no files: claim-level links and rationale live in DB tables, not only saved JSON.
  • Requirement 2 — save state before starting. Per-phase checkpoints so any phase reverts to a known-working state.

Scope of THIS PR: Phases −1 → 2 (pre-flight checkpoints, schema incl. the claim-level provenance tables, fetch dual-write, parse step, --source db extraction) + a runbook for the live 7-day proof. Phase 3 (populate provenance; retire the JSON middle) and Phase 4 (historical backfill) are roadmap, deferred to follow-on PRs. The claim-level schema lands now (cheap, additive); the wiring that fills it is Phase 3.

Three design questions, answered

1 · Visibility into the synthesis run that created a finding? Today run_slug is a free string on findings.first_seen_run / finding_events.run_slug, pointing at a data/runs/<slug>/ folder — seeing the run means opening a file. There is no runs table. The no-files end state needs one: a new synthesis_runs table that run_slug is a (logical) FK to, holding run metadata + the synthesis/adjudication output. finding_events gives the full per-run history.

2 · Is a finding linked with the claim or the article? Today: the article only (findings.evidence = JSON article_ids). Target: the claim is primary — and since claim_id = article_id::claim_N already embeds the article, the article link is a derivable rollup. New finding_claims is the source of truth; findings.evidence stays as a denormalized rollup the existing shortlist_candidates overlap matcher already reads.

3 · Where does the synthesis go? Both already render markdown (deep_synthesis.*.md, adjudication.md). The DB keeps only folded findings/events, never the narrative. Target (no files): persist both rendered .md blobs verbatim into synthesis_runs (synthesis_md / adjudication_md) — markdown with links, not strict JSON columns (freeform, hard to schematize). Structured provenance stays in finding_claims; the human-readable "why" is the markdown blob.

−1 Pre-flight checkpointsSave state for rollback before any phase mutates the DB Do first ▼

Per-phase git checkpoints + a DB snapshot before the migration mutates anything, so any phase reverts to a known-good state independent of the durability decision below.

Decision (recommended) — DB durability: do NOT commit verbatim Layer 0 into git. It holds authenticated full article bodies (ToS/licensing exposure + unbounded repo bloat — every fetch rewrites the blob permanently). Recommended: untrack data/intel.db (gitignore + git rm --cached); durability via per-phase snapshot tarballs + litestream. Alternative: keep a tracked structured DB but route fetch_responses.response_body to a gitignored sidecar store, referenced by id. Either way: no authenticated full-text in git history. Make this call before Phase 1 writes Layer 0.

Added to scripts/intel_db.py following its PRAGMA user_version pattern. New RAW_LAYER_SCHEMA_SQL, all CREATE TABLE IF NOT EXISTS:

  • fetch_responses (Layer 0): id PK, fetch_run_id, url, fetched_at, http_status, response_body, content_type, response_hash, UNIQUE(url, response_hash). response_body nullable + response_body_ref if split to a sidecar; inline if the DB is untracked. Re-parse metadata: feed_kind ('substack'|'extra'|'fulltext') + feed_meta_json. Fulltext rows store no article_id — join key is the GET's URL stored canonicalized in indexed norm_url, resolved later against a map built from both raw_articles.raw_link and .link (each via the same lib.canonicalize_link, so a substack.com raw_link vs custom-domain link can't strand the response). article_id is computed once, at parse, never at fetch.
  • Layer 1 is two tables (teaser + full upgrade are two parses of one article_id, can't share an article_id-PK row): raw_articles (current row, article_id PK, the full parse_rss_items field set + published_at derived via lib.parse_date, winning-version body, current_version_id) and raw_article_versions (append-only, surrogate version_id PK, logical fetch_response_id with sentinel 0 for Tier-B/no-Layer-0, UNIQUE(article_id, fetch_response_id), ON CONFLICT … DO UPDATE so a better re-parse refreshes body in place). Winner picked by body_is_full DESC, body_length DESC, parsed_at DESC.
  • finding_claims (event table): PRIMARY KEY (finding_id, claim_id, run_slug) so per-run rationale isn't clobbered; index on claim_id. Populated in Phase 3.
  • synthesis_runs: run_slug PK, run/window metadata, synthesis_md / adjudication_md blobs. Populated in Phase 3.

Migration: bump SCHEMA_VERSION 3→4, add _migrate_v3_to_v4, chain in connect(). Accessors (no self-commit, caller owns txn): insert_fetch_response, upsert_raw_article_version, load_raw_articles_window, link_finding_claim. All new writes gated behind raw_ingest_enabled().
