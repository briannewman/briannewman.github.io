M1 — Findings Registry
+ Prediction Capture

A SQLite registry that ingests validated synthesis output, matches findings across runs, and feeds open findings back into the prompt as delta-only standing context — while capturing implied predictions.

branch m1-registry 9 tasks · TDD 5 new scripts subagent-driven 2026-06-10
Source: plans/2026-06-10-m1-findings-registry.md

Overview

Architecture
New data/intel.db (SQLite, schema in scripts/intel_db.py) holds findings, append-only status events, predictions, articles, and embeddings. registry_ingest.py runs after synthesis_validator passes and before save_run.py: it extracts findings from the freeform envelope, embeds them (text-embedding-3-small), shortlists registry matches by cosine + evidence overlap, confirms with a cheap LLM call, then records events or inserts new rows. build_standing_context.py renders open findings into the synthesis prompt; the validator gains finding_id membership checking.
Tech stack
Python 3 stdlib (sqlite3, array, hashlib), OpenAI SDK (already a dependency — the batch_extract.py pattern), unittest.
Spec
Target Architecture Sequencing — docs/superpowers/specs/2026-06-10-target-architecture-sequencing-design.md
One deliberate adaptation from the spec. The no-weekly-reruns gate (“near-duplicate of an open finding without a finding_updates reference”) runs as a pre-promotion exit-code gate in registry_ingest.py, not inside the validator's model-retry loop — the validator is pure/mechanical and has no embedding access. Same enforcement point (nothing un-gated reaches save_run), different mechanism.
Post-review deviations (final review, same day). registry_ingest.py refuses to run without an existing data/intel.db (seed first — prevents silently flipping the pipeline into registry mode); finding_updates statuses outside the advertised four (strengthened | weakened | confirmed | refuted) are gated, never coerced; unknown finding_ids in updates gate instead of crashing; seed_registry.py only ingests gpt55/opus lanes by default (--lanes); synthesis-time valid_finding_ids covers ACTIVE findings only. Deferred knowingly: db schema-version stamping in the save_run manifest (do before M2 migrations); standing-context mechanism one-liners (token economy); prediction extraction from delta-only finding_updates.

Context for an engineer new to this repo

  • Scripts live in scripts/, run as python3 scripts/<name>.py, and contain both library functions and a CLI main(). Tests import them via an importlib loader. Run with python3 -m unittest tests.<module> -v from the repo root.
  • The two synthesis lanes produce different freeform JSON envelopes. GPT-5.5: top-level lists like whole_corpus_findings, items have finding/explanation/why_it_matters/evidence. Opus: findings items have title/statement/evidence. The only structural constant (enforced by synthesis_validator.py) is that evidence is a list of dicts each carrying an exact article_id. Finding extraction must be structural, not schema-bound.
  • scripts/lib.py provides: DATA_DIR, WORKING_NEWSLETTERS_PATH, DEFAULT_OPENAI_MODEL ('gpt-5.5'), load_json, newsletter_openai_api_key(), tmp_run_slug(), tmp_artifact_dir(), deep_synthesis_lane_json_paths().
  • The OpenAI structured-output call pattern lives in scripts/batch_extract.py:59-90.
  • data/intel.db is committed to git. A bad ingest is undone with git checkout -- data/intel.db.
