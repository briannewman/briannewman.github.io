Mechanical Synthesis
Output Validator

Move the grounding hard invariants — exact article_id membership, structured references, evidence counts, no prose IDs — from prompt text into a mechanical validator with one corrective retry turn, wired into both synthesis runners.

branch fable 6 tasks · TDD stdlib only gate M1 extends 2026-06-10
Source: plans/2026-06-10-synthesis-output-validator.md

Overview

Architecture
A new pure module scripts/synthesis_validator.py walks any synthesis JSON generically (the two models emit different envelopes), returns a violation list, and provides run_with_validation() — a wrapper that calls a provider closure, parses, validates, and on error-level violations sends ONE corrective retry turn (original prompt + prior output + violations). The validator never mutates model output. Both run_deep_synthesis.py and run_feedback_synthesis.py wrap their existing dispatch in a closure and route it through the wrapper; results land in a .validation.json sidecar and the .meta.json.
Tech stack
Python 3 stdlib only. Tests use unittest with the repo's load_module convention.
Relationship
This validator is the gate the M1 registry extends (it adds finding_id membership in Task 6 of that plan). Ship this first.
The HANDOVER.md “Hard invariants” section is the spec. Every article reference must be a structured object with an exact article_id from the corpus; ≤3 evidence refs per finding (4 allowed only when genuinely needed — mechanically: error above 4, warning at exactly 4); valid JSON envelope. Validation must be at least as strict after any refactor.

Context for the implementer

  • Repo: /Users/briannewman/Projects/newsletter-intel-fable, branch fable. Run all commands from the repo root.
  • Real output shapes (verified 2026-06-10): Opus emits {overall, findings[{title, explanation, evidence[{article_id, title, role}]}], corpus_questions[...]}; GPT-5.5 emits {corpus_window, article_links, whole_corpus_patterns, corpus_questions}. Never assume top-level key names — walk the whole tree.
  • ID-bearing keys are exactly article_id, article_id_a, article_id_b (scripts/synthesis_evidence.py:21). Article IDs look like article-<20 hex chars> (scripts/lib.py:593-606).
  • extract_json_object (scripts/lib.py:616-624) is the existing parse step; it raises RuntimeError when no JSON object is found.
  • Do NOT modify lib.py provider runners, the prompts (DEEP_SYNTHESIS_PROMPT.md, FEEDBACK_SYNTHESIS_PROMPT.md), or the system prompts. The prompt-side grounding rules stay; this adds the code-side guarantee.

Out of scope (deliberately)

Pass/fail eval gates (cross-genre count, high-frequency-token headline check, prior-window repeat detection) — separate backlog item.
Fixed output envelope schema — prompts backlog; the validator is envelope-agnostic by design so it keeps working when the envelope is later fixed.
Auto-repair / mutation of model output — forbidden; the validator only reports and retries.
Provider prompt caching for cheaper retries — separate backlog item (cache-friendly ordering).
run_adjudication.py — already has its own deterministic citation enforcement.
