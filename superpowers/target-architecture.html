Why this build order

M1 is the only milestone that improves output quality at the very next run with zero prerequisites, and it starts the prediction clock — resolutions need calendar time, so capturing predictions early is the only way M5 has anything to grade.

M2 (prompt redesign plus paid re-extract of the full corpus) comes after the quick win. M3 and M4 depend on M2 embeddings. Deferring author calibration costs nothing: resolution events are stored durably; calibration is a derived view computable retroactively.

Storage — data/intel.db

One SQLite database via scripts/intel_db.py (schema, migrations, accessors). Run artifacts stay flat JSON as today; intel.db holds only cross-run state. save_run.py manifest records db schema version; the db file is committed after each promoted run.

Milestones M1–M5

M1 — Findings registry + prediction capture

Runs after validator passes, before save_run.py. Injects standing context into synthesis; starts the prediction clock.

  • registry_ingest.py — embed findings, shortlist matches, Haiku-class confirm, events or new rows; extract falsifiable predictions
  • build_standing_context.py — open findings block in stable prompt prefix (delta-only instruction)
  • seed_registry.py — bootstrap from data/runs/; stale findings older than two windows unless re-cited

Touch-points: optional finding_updates on synthesis envelope; validator checks finding_id; near-duplicate gate in registry ingest (pre-promotion exit code).

M2 — Digest redesign + full corpus re-extract

Claims become rich objects; paid one-shot re-extract ~739 articles; retires legacy synthesis payload prunes.

  • Schema: text, quote, kind, entities, quantities, mechanism_chain, caveats, novelty; article content_type
  • check_entailment.py — quote entails claim text; failures flagged unentailed
  • embed_claims.py — embeddings for claims and digests
  • Every kind=prediction claim → predictions row

Regression: golden-window synthesis before/after. Copy golden fixture out of gitignored grok worktree before M2 starts.

M3 — Incremental graph + liveness + advisory trigger

  • update_graph.py — per new article: entailment → embed → score vs stored embeddings; weighted blend + time decay + bridge inversion; top-K≈5 edges
  • Liveness is computed: in window ∨ new edge in last N ingests ∨ cited by open finding ∨ open prediction
  • Advisory trigger after fetch: sum new-edge weight vs threshold T; print top 5 edges — human still runs synthesis

M4 — Recall-set synthesis + escalation verification

  • Recall stage in run_deep_synthesis.py: ~15 prior-window digests linked by strong edges, marked recall block
  • Enrichment over window + recall; validator widens ID set only
  • Post-validation grounding on cited articles (bodies in article_bodies.json); failures flagged, never truncated

M5 — Prediction resolution grader + write-back

  • resolve_predictions.py — corpus search first (embedding), web fallback; adjudication-style citation attachment by code
  • Default verdict unresolvable; explicit outcome required for confirm/refute
  • Write-backs to predictions, articles, finding events — durable for retroactive calibration
  • Hindsight firewall: graph and enrichment never read outcome columns

Where this sits in the refresh pipeline

Guardrails and testing

◇
Golden windowEach milestone shippable independently; same fixture before/after; finding quality diffed by hand (user is taste arbiter).
◇
Invariants only extendfinding_id membership (M1), recall-set ID widening (M4); grounding identical for recalled articles.
◇
Registry decouplingRegistry consumes validator-enforced envelope only — M2 digest changes cannot break M1.
◇
Unit testsFinding match pairs, edge scoring, liveness predicate, resolution conservatism (unresolvable default).

Explicitly out of scope

Author-calibration weighting in synthesis; refuted-findings-as-negative-examples; scheduled automation; vector databases; per-author standing-position modeling beyond M2 novelty lookup.
